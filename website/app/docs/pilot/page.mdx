# Pilot mode

`wingman pilot run "<goal>"` plans a multi-task piece of work, validates the
task dependency graph, spawns specialised worker agents in isolated git
worktrees, and converges their output into a single PR.

The plan is a DAG: before any worker is dispatched, the scheduler rejects a
task that depends on itself, a dep pointing at an unknown id, or a dependency
cycle. The check runs again on every manager- or splitter-issued edge, so a bad
edge can never wedge a run on a graph the scheduler would wait on forever — it
fails fast instead.

## Capability tiers

```text
assist     You approve every decision. Agent plans, you confirm, agent executes
           one run, opens a PR, exits. No daemon, no critic, no learning.
copilot    Default. Agent flies; you monitor and intervene at decision points.
           Trust-tiered approval, self-healing retries, per-task reviewer,
           real verification, PR automation, cross-run learning.
autopilot  Agent flies and navigates. Daemon mode, multi-channel intake,
           critic agent, knowledge graph, tool synthesis, sandboxed execution.
```

Pick a tier in `~/.wingman/config.toml`:

```toml
[pilot]
tier                  = "copilot"
default_model         = "anthropic/claude-opus-4-8"   # manager + reviewers
worker_model          = "anthropic/claude-haiku-4-5"  # workers
max_concurrent_agents = 4
max_usd               = 10.0
task_timeout_secs     = 1800
```

## Quick start

```bash
# One-shot: plan, approve, spawn workers, open PR
wingman pilot run "add a --version-only flag to wingman-cli"

# Plan only — write tasks.jsonl and exit
wingman pilot run --plan-only "<goal>"

# Auto-approve the plan (skip the y/e/n gate)
wingman pilot run --yes "<goal>"

# Dashboard
wingman pilot status              # one-shot summary of the latest run
wingman pilot watch               # live ASCII dashboard, polls state.json
wingman pilot watch <run-id>      # specific run
```

## Per-run artefacts

Per-run artefacts land under `<project>/.wingman/autonomous/<run-id>/`:

```text
<run-id>/
  tasks.jsonl   # append-only event log
  state.json    # latest snapshot (rewritten after every event)
```

## Token accounting

At the end of a run, Pilot prints a per-phase token breakdown — planner,
manager, reviewer, critic, and each worker — bucketed from the run's `agent.usd`
events and sorted by total tokens. The manager loop's own usage is captured
rather than dropped, so the totals reflect every LLM pass the run made. One-shot
passes (planner, reviewer) set an `AfterSystem` cache breakpoint so the reused
system prompt is read back from cache on the second call instead of re-billed.

## Status

M1 is shipped: RunStore, planner with DAG validation, worker subprocess with a
cross-platform supervisor, manager + orchestrator, git worktrees +
squash-merge, `gh` PR creation, the live dashboard, cost-cap enforcement, and
the provider-support gate. End-to-end runs need real API keys and are
user-validated. The higher tiers layer in M2/M3 machinery (critic, sandbox,
knowledge graph, tool synthesis, multi-channel intake, daemon) — scaffolded in
`wingman-autonomous` and rolling out per tier. See
[pilot provider support](/docs/pilot-providers) for which backends can drive it.
